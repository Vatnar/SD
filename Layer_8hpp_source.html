12// TODO: BIG REFACTOR - Decouple Rendering from Layers
13// The current design tightly couples rendering logic (Vulkan commands, pipeline management, etc.)
14// with the Layer system. This makes layers "heavy" and less modular. A better approach would be to
15// move towards a more data-oriented design where layers primarily manage game logic and scene data,
16// and a separate rendering system handles the drawing.
17//
18// How to do it:
19// 1. Create a central "Renderer" or "RenderSystem" class. This class would manage the main command
20// buffers, render passes, and frame submission.
21// 2. Layers should not directly create or manage Vulkan resources like pipelines, descriptor sets,
22// or command buffers.
23// 3. Instead of `RecordCommands`, layers could have a method like
24// `OnCollectRenderables(RenderQueue& queue)` or `Submit(Renderer& renderer)`.
25// 4. Inside this method, layers would submit "Renderables" or "DrawCommands" to the renderer. These
26// are simple data structures
27// containing information needed for drawing (e.g., mesh, material/shader, transform, textures).
28// 5. The Renderer would then process this queue of renderables, sort them (e.g., by pipeline,
29// material, or depth), and record the necessary Vulkan commands.
30//
31// Example:
32// class Layer {
33// ...
34// virtual void OnUpdate(float dt) {
35// // Update game logic, transforms, etc.
36// }
37// virtual void Submit(Renderer& renderer) {
38// renderer.Submit(m_MyRenderableObject); // m_MyRenderableObject contains mesh, material,
39// transform
40// }
41// ...
42// };
43//
44// This would make layers much thinner and focused on layer logic, while centralizing the complex
45// rendering logic in one place.
